# [쉬운주차장] 구글 서치콘솔 '가져올 수 없음' 트러블슈팅 #5

## 1. 현상: 특정 사이트맵만 수집 실패

사이트맵 인덱스(`sitemap.xml`)를 제출하고 결과를 확인했는데 상황이 이상했습니다. `sitemap-0.xml`은 정상적으로 가져갔는데, `1, 2, 5`번 파일에서 '가져올 수 없음' 에러가 뜸. 개별 파일이 실패하니 인덱스 파일 전체도 오류 상태가 됐습니다.

## 2. 조사: 모든 검증기는 '정상'이었음

가장 먼저 의심한 건 사이트맵 내부의 '불량 URL'이나 XML 문법 오류. 근데 확인해보니 다 정상이었음.

- **전수 조사:** `sitemap-5.xml`에 포함된 5,000개 URL을 스크립트로 전수 검사했습니다. 5,000개 전부 `200 OK`.
- **외부 검증:** `xml-sitemaps.com`이랑 `nuxtseo.com` 사이트맵 검증 도구를 돌려봤는데, 문법적으로 문제없다는 판정.
- **수동 확인:** 로컬에서 `curl`로 직접 호출해도 1MB 내외 XML이 1초 안에 정상적으로 내려옴.

데이터, URL, XML 구조 전부 문제없음. 모든 도구가 '초록불'인데 구글 서치콘솔만 '빨간불'인 상황이었습니다.

## 3. 그래서 원인이 뭔데?

솔직히 아직 모릅니다. 브라우저에서 직접 URL을 치면 잘 나옴. `curl`로 호출해도 1초 안에 내려옴. 근데 구글만 못 가져감.

처음에는 Cloudflare Workers의 **50ms CPU 시간 제한**을 의심했음. 현재 사이트맵 생성 로직이 이렇게 생겼는데:

```javascript
let xml = '<?xml version="1.0" encoding="UTF-8"?><urlset ...>';
const rows = await db.prepare("SELECT ...").bind(offset, 5000).all();

for (const row of rows.results) {
  const slug = generateSlug(row.name, row.city, row.district);
  xml += `<url><loc>https://easyparking.day/lot/${slug}</loc></url>`;
}
```

5,000개 데이터를 루프 돌면서 `+=`로 XML 문자열을 이어 붙이는 방식임. JS에서 문자열 `+=`는 데이터가 많아질수록 메모리 재할당 비용이 급격히 커져서, CPU 50ms 제한에 걸릴 수 있다고 생각했습니다.

근데 직접 접속하면 잘 되니까 이것만으로는 설명이 안 됨. 구글봇이 동시에 여러 파일을 요청할 때만 터지는 건지, 아니면 아예 다른 문제인지 현재로서는 확신이 없습니다.

## 4. 왜 0번만 성공했을까?

0번부터 6번까지 데이터 개수는 전부 5,000개로 동일함. 근데 왜 0번만 성공하고 1, 2, 5번은 실패했을까.

주차장 이름 길이나 유니코드 문자 차이로 연산 시간이 미세하게 달라졌을 거라고 봅니다. 50ms 한계선 위에서는 몇 밀리초 차이가 '성공'과 '가져올 수 없음'을 가름. 개별 URL 문제가 아니라 응답 자체가 안정적이지 않은 상태였음.

## 5. 아직 의심 가는 것들

CPU 타임아웃이 유력하긴 한데, 아직 추측 단계임. 다음에 확인해봐야 할 게 세 가지 남아있습니다.

- **fetching 속도:** 각 사이트맵 파일의 실제 응답 시간을 재봐야 함. 파일별로 편차가 있는지, 구글봇이 타임아웃 처리하는 기준에 걸리는지 확인.
- **URL 접근 가능 여부:** 스크립트로는 전부 `200 OK`였지만, 구글봇 관점에서 실제로 접근 가능한지 다시 체크. 슬러그 생성 로직에서 이상한 URL이 만들어지고 있을 수도 있음.
- **gzip 압축:** 현재 XML을 그대로 내려주고 있는데, `sitemap.xml.gz`로 압축해서 제공하면 전송 크기가 줄어들어 fetching 자체가 빨라질 수 있음.

이걸로도 안 되면 솔직히 트래픽 문제일 수도 있습니다. 서비스 초기라 구글이 크롤링 우선순위를 낮게 잡고 있을 수 있음. 그때는 트래픽이 좀 더 쌓이고 나서 사이트맵을 다시 제출해봐야 할 것 같습니다.
