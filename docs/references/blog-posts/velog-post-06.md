# [쉬운주차장] 블로그 크롤링으로 3만 5천 개 주차장 난이도를 매기기까지 #6

## 1. 왜 크롤링이 필요했나

쉬운주차장의 핵심 기능은 주차장별 "난이도"입니다. 좁은지, 기계식인지, 초보가 갈 만한지. 처음엔 사용자 리뷰의 별점 평균으로 난이도를 보여줬는데, 문제가 바로 드러났습니다.

**리뷰가 없습니다.**

3만 5천 개 주차장 중 사용자 리뷰가 달린 곳은 손에 꼽힘. 리뷰 1건에 5점짜리가 리뷰 50건 평균 4.2점보다 높게 나오는 문제도 있었고요. 그래서 블로그/카페/유튜브 댓글 같은 웹 콘텐츠를 크롤링해서 난이도 점수에 반영하기로 했습니다.

근데 크롤링한 데이터를 바로 점수에 반영할 수는 없습니다. "강남역 맛집 추천! 주차 가능" 같은 블로그 글이 주차 난이도에 대한 정보를 담고 있을까요? 광고는? 부동산 매물 글은?

결국 **수집 → 필터링 → 주차장 매칭 → 점수 계산**이라는 4단계 파이프라인을 만들게 됐습니다.

---

## 2. 아키텍처: 2-테이블 구조

처음엔 크롤링 결과를 `web_sources` 테이블 하나로 관리했습니다. 근데 금방 한계가 옴.

- 크롤링 원본과 검증된 데이터가 섞여있어서 어디까지 처리했는지 추적이 안 됨
- 필터에서 탈락한 건인지, 아직 필터를 안 돌린 건인지 구분이 안 됨
- 주차장 매칭 전 데이터와 후 데이터가 같은 테이블에 있으니 쿼리도 복잡해짐

그래서 테이블을 둘로 쪼갰습니다.

```
web_sources_raw (크롤링 원본 + 파이프라인 상태)
  ↓ AI 필터 통과 + 주차장 매칭 완료
web_sources (검증된 데이터만)
```

`web_sources_raw`에는 크롤링한 모든 것을 때려넣고, `filter_passed`, `matched_at` 같은 상태 컬럼으로 파이프라인 진행도를 추적합니다. `web_sources`에는 최종 검증을 통과한 데이터만 들어감. 점수 계산은 `web_sources`만 바라봅니다.

---

## 3. 1단계: 크롤링 — 네이버, 유튜브, 검색엔진

수집 소스는 4개입니다.

| 크롤러 | 소스 | 주기 |
|--------|------|------|
| 네이버 블로그/카페 | Naver Search API | 매시간 |
| YouTube | YouTube Data API | 매시간 |
| Brave Search | Brave Search API | 일 1회 |
| DuckDuckGo | Crawl4AI 온프레미스 | 매시 30분 (별도 cron) |

전부 Cloudflare Workers의 Cron Trigger로 매시간 자동 실행됩니다. `scheduled.ts` 하나에서 오케스트레이션하고, 각 크롤러는 독립 모듈.

DuckDuckGo만 별도 cron으로 돌리는데, 이유는 **Cloudflare Workers의 subrequest 한도** 때문입니다. 메인 파이프라인에서 네이버+유튜브+Brave+AI필터+매칭까지 돌리면 subrequest가 빡빡해서, DDG를 매시 30분으로 분리했습니다.

DDG 크롤러는 좀 특이한데, DuckDuckGo에는 공식 API가 없어서 [Crawl4AI](https://github.com/unclecode/crawl4ai)라는 오픈소스 웹 크롤러를 온프레미스에 띄워서 프록시처럼 사용합니다. Workers에서 Crawl4AI로 검색어를 보내면, Crawl4AI가 DDG 검색 결과 페이지를 파싱해서 돌려주는 구조.

모든 크롤러는 `web_sources_raw`에 `INSERT OR IGNORE`로 저장합니다. `source_id`(소스 플랫폼 + 고유 ID) 기준 중복 방지. 이 단계에서는 필터링이나 매칭 없이 원본만 쌓습니다.

## 4. 2단계: AI 필터 — Haiku가 광고를 걸러낸다

크롤링하면 온갖 것들이 들어옵니다. 부동산 분양 글, 결혼식장 주차 안내, 월주차 요금 광고... 주차 난이도와는 아무 관계 없는 것들.

처음에는 키워드 기반 필터를 시도했습니다. `is_ad` 플래그, 제외 키워드("분양", "월주차", "웨딩") 같은 룰을 만들었는데, 500건 파일럿을 돌려보니 **키워드 필터가 유효한 주차 글 10건을 광고로 오분류**하는 문제가 나옴. "이 주차장 월주차도 가능한데 방문 주차도 됩니다" 같은 글이 "월주차" 키워드에 걸려서 통째로 버려지는 식.

그래서 **Claude Haiku에 필터링을 맡겼습니다.**

```typescript
// ai-filter.ts — Haiku에게 보내는 시스템 프롬프트 (핵심부)
const SYSTEM_PROMPT = `주차장 검색 결과를 분류하는 JSON 분류기입니다.

판단 기준:
- filter_passed: 주차장에 대한 유용한 정보가 있으면 true
  - "ad": 제품/서비스 홍보 → false
  - "realestate": 부동산 분양/매매 → false
  - "irrelevant": 주차와 완전히 무관 → false
- difficulty_keywords: 난이도 관련 표현 추출
- sentiment_score: 초보 운전자 관점 주차 용이성 (1.0~5.0)
- summary: 핵심 한줄 (20자 이내)`;
```

10건씩 묶어서 배치로 보내고, 5개 배치를 병렬 처리합니다. 1회 cron에 최대 100건. Cloudflare Workers Free plan의 30초 wall time 제한 안에서 최대한 뽑아내려고 이것저것 튜닝한 결과물입니다.

```
10건 배치 × 5 병렬 = 50건/라운드
2 라운드 = 100건/cron
```

처음에는 배치 사이즈를 200건으로 올렸다가, 50건으로 줄였다가, 결국 100건에 정착. 밀린 26,000건을 처리하느라 배치 사이즈를 몇 번 바꿨는데, Free plan 30초 제한이 생각보다 빡빡합니다.

## 5. 3단계: 주차장 매칭 — FTS + AI 하이브리드

AI 필터를 통과한 블로그 글이 "어떤 주차장"에 대한 글인지 연결해야 합니다. 블로그 제목이 "강남역 주차 꿀팁"이면, 우리 DB에 있는 3만 5천 개 주차장 중 어디인지 찾아야 함.

이걸 **하이브리드 매칭**으로 풀었습니다.

**Step 1: 키워드 추출**

블로그 제목+본문에서 불용어("주차장", "후기", "추천", "네이버"...)를 제거하고, 의미 있는 키워드 최대 5개를 뽑습니다.

```typescript
const STOP_WORDS = new Set([
  '주차장', '주차', '후기', '정보', '추천', '요금',
  '무료', '네이버', '블로그', '카페', '유튜브', ...
]);
```

**Step 2: FTS5 검색**

SQLite Full-Text Search로 주차장 후보를 최대 20개 뽑고, FTS 결과가 3개 미만이면 LIKE 폴백.

**Step 3: 신뢰도 판정**

후보별로 제목/본문과의 매칭 신뢰도를 `high / medium / low`로 판정합니다.

```
high   → 바로 web_sources INSERT (AI 불필요)
medium → Haiku AI 검증 후 INSERT
low    → 스킵
```

초기 파일럿에서 키워드 매칭만 쓰면 정밀도가 30~40%였습니다. 블로그당 4.1개 주차장이 매칭되는데, 대부분 오매칭. AI 매칭으로 바꾸니 블로그당 1.6개, 정밀도 ~95%.

하지만 모든 건에 AI를 돌리면 비용이 너무 나가니까, 신뢰도가 높은 건은 AI 없이 바로 통과시키고, 애매한 건만 AI한테 물어보는 하이브리드 방식으로 정착했습니다.

## 6. 4단계: 베이지안 스코어링

매칭이 끝나면 해당 주차장의 난이도 점수를 재계산합니다. 여기서 **베이지안 평균**을 씁니다.

리뷰 1건에 5점짜리가 50건 평균 4.2보다 높게 나오는 문제, 기억하시죠? 베이지안 평균은 이걸 해결합니다.

```
Score = (C × prior + n_eff × raw_score) / (C + n_eff)
```

- `C = 5`: 유효 5건 이상이면 실제 데이터가 지배
- `prior`: 구조 속성 기반 사전 점수 (기계식 → 감점, 200면 이상 → 가점 등)
- `n_eff`: 유효 데이터량 (사용자 리뷰 1건=1.0, 블로그 1건=0.2)

데이터가 없으면 구조 속성 기반 추정치가 나오고, 데이터가 쌓일수록 실제 평균으로 수렴합니다.

| 상황 | n_eff | 결과 |
|------|-------|------|
| 데이터 없음 | 0 | 구조 속성 기반 추정 (3.0 ± 보정) |
| 리뷰 1건 5점 | 1.0 | **3.33** (과대평가 억제) |
| 리뷰 50건 평균 4.2 | 50.0 | **4.09** (실제 평균 수렴) |

스코어링은 전체 배치를 돌리지 않고, **최근 2시간 내 새로 매칭된 주차장만** 재계산합니다. `crawl_progress` 테이블에 마지막 실행 시각을 기록해두고, 그 이후에 매칭된 건만 대상으로 잡음. 3만 5천 개 전체를 매시간 재계산하면 Workers CPU 시간이 터지니까요.

## 7. 파이프라인 운영에서 배운 것들

### Workers Free Plan 30초의 벽

Cloudflare Workers Free plan은 wall time 30초 제한이 있습니다. AI 필터에서 Haiku API를 호출하고, 매칭에서도 Haiku를 쓰니까, 한 cron에서 처리할 수 있는 양이 제한됨.

처음에 배치 사이즈 200으로 잡았다가 타임아웃, 50으로 줄였다가 처리량 부족, 결국 "10건 배치 × 5 병렬 × 2라운드 = 100건"에 안착. 밀린 데이터 26,000건을 소화하는 데 며칠 걸렸습니다.

### is_ad 컬럼의 죽음

키워드 필터로 광고를 잡으려고 `is_ad` 컬럼을 만들었는데, 파일럿에서 false negative가 10건 나오자마자 버렸습니다. AI가 맥락을 보고 판단하는 게 단순 키워드 매칭보다 월등히 정확함. 결국 `is_ad` 컬럼 자체를 DROP하고 AI 필터로 완전 대체.

### DDG를 별도 cron으로 분리한 이유

메인 cron에 DuckDuckGo 크롤러를 넣으면 subrequest 한도(Workers Free plan 50개)에 걸립니다. 네이버+유튜브+Brave+AI필터+매칭이 이미 subrequest를 꽤 쓰고 있어서, DDG를 매시 30분에 따로 돌리기로 했음. cron 하나 추가하는 게 코드 복잡도를 올리는 것보다 나았습니다.

## 8. 현재 상태

이 파이프라인이 매시간 자동으로 돌면서:

1. 네이버/유튜브/Brave/DDG에서 주차장 관련 콘텐츠를 수집하고
2. Haiku가 광고/무관 콘텐츠를 걸러내고
3. FTS + AI로 주차장에 매칭하고
4. 베이지안 평균으로 난이도 점수를 갱신합니다

사용자 리뷰가 없어도 블로그/유튜브 데이터로 난이도를 보여줄 수 있게 됐습니다. "리뷰 없음"이던 주차장에 "추정 3.2 (블로그 기반)" 같은 점수가 붙기 시작함.

다음 포스팅에서는 이 파이프라인 위에 "주변 갈만한 곳" 큐레이션을 얹은 작업을 공유하겠습니다.
