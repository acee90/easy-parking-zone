# 주차장 기본정보 보강 전략

작성일: 2026-04-24

## 목적

`parking_lots`의 기본정보 결손을 줄여 상세 페이지 품질과 검색 가치, 사용자 신뢰도를 높인다.

대상 기본정보:

- 운영시간
- 요금
- 주차면수
- 전화번호
- 특기사항

## 현재 진단

로컬 D1 기준 전체 `parking_lots`는 31,939건이다.

핵심 결손 현황:

- 면수 누락: 15,960건 (49.97%)
- 운영시간 누락: 15,676건 (49.08%)
- 유료 주차장 중 기본 요금 누락: 7,901건
- 면수/운영시간/요금 중 하나라도 누락: 17,334건 (54.27%)

결손의 대부분은 `KA-*` 소스에 몰려 있다.

- `KA-*`: 19,268건
- `NV-*`: 892건
- 공공데이터 ID: 11,779건

특히 `KA-*`에서 운영시간과 면수 누락이 집중적으로 발생한다.

## 공공데이터 활용 한계

공공데이터 원본 자체의 품질은 좋다.

- 공공데이터 적재본: 11,779건
- core 정보 완성형: 11,432건 (97.05%)
- 운영시간 누락: 0건
- 면수 누락: 2건

문제는 원본 품질이 아니라 `KA/NV ↔ 공공데이터` 매칭 범위다.

현재 규칙(150m + 이름 유사도 0.5 이상)으로 매칭 가능한 `KA/NV`는 제한적이다.

- `KA/NV` 총합: 20,160건
- 공공데이터 매칭 가능: 1,280건 (6.35%)
- 공공데이터 core 정보까지 충분한 매칭: 1,246건

즉 남아 있는 대규모 결손은 공공데이터만으로 해결하기 어렵다.

## 공공데이터 매칭 실패 상위 패턴

`KA-*` 기준 상위 실패 패턴:

1. 공공데이터 미수록 또는 좌표 불일치 추정
2. 거리권 내 후보는 있으나 이름 유사도 부족
3. 300m 내 공공데이터는 있으나 이름도 불일치
4. 공영형 명칭이지만 공공데이터 근접 후보 없음
5. 사설/건물 부설형 추정

해석:

- 관광지/상업시설/호텔/마트/병원 등은 공공데이터 미수록 가능성이 높다.
- 공영주차장도 이름 체계가 달라 자동 매칭이 실패하는 경우가 있다.
- 일부는 반경 완화나 alias 규칙으로 회수 가능하지만, 전체 문제를 뒤집을 수준은 아니다.

## 샘플 검증 결과

### 카카오 상세 보강 샘플 20건

조건:

- `KA-*`
- `노외`
- 면수/운영시간/요금이 동시에 비어 있거나 결손이 큰 항목
- 서울/경기/인천/부산/대구 도시권 우선

`scripts/enrich-kakao-place.ts --dry-run` 결과:

- 대상: 20건
- 업데이트 가능: 10건
- 성공률: 50%

필드별 성과:

- 운영시간 보강: 10건
- 요금 보강: 6건
- 면수 보강: 0건

결론:

- 카카오 상세는 운영시간 보강에 유효하다.
- 요금은 일부만 채워진다.
- 면수 보강에는 거의 기여하지 못한다.

## 기본 원칙

1. 필드별로 가장 강한 소스를 다르게 쓴다.
2. 면수와 운영시간/요금을 같은 파이프라인으로 보지 않는다.
3. 구조화 데이터 소스를 우선하고, 블로그는 보조 근거로만 쓴다.
4. 빈 필드 채우기와 기존 값 교정은 분리한다.
5. 보강 출처와 반영 시각은 남긴다.

## 필드별 보강 전략

### 1. 운영시간

우선순위:

1. 카카오 상세
2. 네이버 플레이스 상세
3. 공공데이터
4. 블로그/카페 AI 추출

전략:

- `KA-*` 중 운영시간 누락 항목에 카카오 상세 스크래핑을 우선 적용한다.
- 카카오 상세에서 운영시간이 비어 있으면 네이버 플레이스 상세를 2차로 시도한다.
- 블로그/카페는 구조화 소스가 없을 때만 보조로 쓴다.

적용 규칙:

- 시작/종료 시간이 명확할 때만 반영
- 24시간은 `00:00 ~ 24:00` 또는 내부 규칙에 맞는 형식으로 통일
- 블로그 단일 출처는 `notes` 우선, canonical 필드 반영은 보수적으로 수행

### 2. 요금

우선순위:

1. 카카오 상세
2. 네이버 플레이스 상세
3. 공공데이터
4. 블로그/카페 AI 추출

전략:

- 기본/추가/일최대가 구조화로 보이는 경우만 반영한다.
- `0원 + 무료쿠폰`처럼 UI 해석이 필요한 경우는 별도 규칙을 둔다.
- 블로그 기반 요금 추출은 교차 근거가 없으면 `notes`로만 남긴다.

적용 규칙:

- `base_time`, `base_fee`, `extra_time`, `extra_fee`, `daily_max`를 세트로 본다.
- 일부 값만 있는 경우는 부분 업데이트 가능하되, 기존 유효값을 덮어쓰지 않는다.
- 무료 여부는 명시 근거가 있을 때만 `is_free = 1`로 바꾼다.

### 3. 주차면수

우선순위:

1. 공공데이터
2. 별도 구조화 원천
3. 블로그/플레이스 텍스트 추출

중요:

- 면수는 카카오 상세에서 거의 안 나온다.
- 현재 가장 큰 결손 중 하나지만 카카오 보강 파이프라인으로 해결되지 않는다.

전략:

- 면수는 별도 트랙으로 관리한다.
- 공공데이터 매칭으로 채울 수 있는 건 먼저 회수한다.
- 이후 네이버 플레이스/공식 안내문/시설 페이지 등 구조화에 가까운 원천을 탐색한다.
- 블로그 텍스트에서 추출한 면수는 신뢰도 낮음으로 보고 바로 canonical 반영하지 않는다.

## 소스별 역할 정의

### 공공데이터

- 강점: 운영시간, 요금, 면수의 구조화 품질이 높음
- 약점: 커버리지가 제한적이고 `KA/NV`와 자동 매칭률이 낮음
- 역할: 정본 역할, 매칭 가능한 공영/공공성 주차장 보강

### 카카오 상세

- 강점: 운영시간, 요금, 무료 여부
- 약점: 면수 없음 또는 매우 희박
- 역할: `KA-*` 운영시간/요금 보강의 1차 수단

### 네이버 플레이스 상세

- 강점: 운영시간, 전화번호, 할인/구매연동/휴무 메모
- 약점: 파싱 난이도, 구조 편차
- 역할: 카카오 실패분의 2차 보강

### 블로그/카페

- 강점: 실제 이용 팁, 혼잡도, 할인 조건, 운영 특이사항
- 약점: 기본정보 정확도 변동 큼
- 역할: `notes`, `aiSummary`, `aiTip*` 및 보조 추출

### DuckDuckGo 검색

- 강점: 공식 홈페이지, 시설 안내 페이지, 병원/호텔/쇼핑몰 안내문 같은 추가 웹 문서를 발견하는 데 유리
- 약점: 구조화 정보보다 텍스트 파편이 많고, 검색 품질 편차가 크다
- 역할: 전체 구조화 소스를 다 돈 뒤에도 남는 잔여 주차장의 마지막 보조 탐색 소스

원칙:

- DDG는 canonical 필드의 1차 근거로 사용하지 않는다.
- DDG는 `카카오 상세 -> 네이버 플레이스 상세`를 모두 돈 뒤 마지막 단계에서만 쓴다.
- DDG 결과는 우선 `notes`, `aiSummary`, `aiTip*` 보강에 사용한다.
- 운영시간/요금 같은 핵심 필드는 공식 안내 페이지가 확인될 때만 제한적으로 반영한다.
- 면수 보강은 DDG 단독 근거로 반영하지 않는다.

## 우선순위 운영

### 현 시점 우선순위

1. `KA-*` + `노외` + 운영시간 누락
2. 도시권 우선
   - 서울
   - 경기
   - 인천
   - 부산
   - 대구
3. 그 다음 결손량 큰 지역
   - 경북
   - 강원특별자치도
   - 경남
   - 전남
   - 충남

### GA 기반 우선순위

향후 GA 데이터를 붙일 수 있으면 우선순위는 다음으로 바꾼다.

우선 정렬 기준:

1. `/wiki/*` 페이지뷰
2. landing page 유입
3. active users
4. 기본정보 결손 개수

즉 "트래픽이 높은데 기본정보가 빈 페이지"를 먼저 고친다.

현재 워크스페이스에는 GA 스크립트는 삽입되어 있으나, API 자격증명이나 export 데이터는 없다. 따라서 당장은 결손량 기준 우선순위를 사용한다.

## 실행 계획

### Phase 1. 카카오 상세 1차 배치

대상:

- `KA-*`
- `노외`
- 운영시간 또는 요금이 비어 있는 항목

방법:

- `scripts/enrich-kakao-place.ts` 배치 실행
- `total_spaces`는 제외하고 `운영시간`, `요금`, `전화`만 대상으로 본다.
- 먼저 `100건 dry-run -> 300건 remote -> 500건 remote` 순으로 확장한다.

목표:

- `KA-*` 잔여 결손의 가장 큰 비중을 구조화 소스로 먼저 줄인다.

### Phase 2. 네이버 플레이스 2차 배치

대상:

- `KA-*`
- 카카오 상세에서 운영시간/요금/전화가 채워지지 않은 잔여 항목

방법:

- 네이버 플레이스 상세 전용 스크립트를 추가한다.
- 카카오 실패분만 대상으로 운영시간/요금/전화 보강을 시도한다.
- 기존 유효값은 덮어쓰지 않고 빈 필드만 채운다.

### Phase 3. 면수 전용 보강

대상:

- `total_spaces <= 0`

방법:

- 공공데이터 매칭은 이미 낮은 커버리지로 확인됐으므로 주력 경로에서 제외한다.
- 공공데이터는 기존 매칭 성공분의 품질 확인과 명확한 alias 보정 케이스에만 제한적으로 사용한다.
- 추가 구조화 원천을 별도로 발굴한다.
- 텍스트 기반 추출은 보수적으로 별도 검증 후 반영한다.

### Phase 4. DDG 최종 잔여 보강

대상:

- 카카오 상세, 네이버 플레이스 상세까지 모두 실패한 항목
- 브랜드/대형 민영 중심의 최종 잔여 항목

방법:

- DDG 검색으로 공식 안내 페이지, 시설 이용안내, 공지문 탐색
- 구조화 필드보다 `notes`와 요약 보강에 우선 사용
- 운영시간/요금은 공식 근거가 확인된 경우에만 제한적으로 반영
- `24시간`, `무료`, `면수`처럼 오탐 리스크가 큰 값은 더 보수적으로 다룬다.

## 스크립트 운영 방향

### 유지

- `scripts/enrich-from-public-data.ts`
- `scripts/enrich-kakao-place.ts`
- `scripts/enrich-from-blog-search.ts`
- `scripts/enrich-targeted.ts`

### 개선 필요

1. 카카오 상세 결과에 필드별 성공 통계 출력
2. 네이버 플레이스 상세 전용 보강 스크립트 추가
3. 보강 대상 추출 쿼리 표준화
4. 면수 보강 트랙 분리
5. 결과 리포트 CSV/JSON 자동 생성
6. DDG 결과를 최종 잔여 주차장 보조 근거용으로 라우팅
7. 공공데이터 경로는 재실행 계획이 아니라 과거 시도/제한적 검증 경로로 명확히 분리

## 운영상 주의점

1. `verified_source`는 마지막 보강 출처만 남기므로, 장기적으로는 필드별 provenance 저장이 필요하다.
2. 무료/유료 판정은 UI 문구 해석 오류 가능성이 있어 보수적으로 반영해야 한다.
3. 블로그 기반 추출값은 canonical 필드보다 `notes`와 AI 요약 보강에 더 적합하다.
4. 같은 실제 주차장이 소스별 row로 분리되어 있어, 장기적으로는 canonical/source 분리 설계가 필요하다.

## 결론

현재 기본정보 보강에서 가장 현실적인 전략은 다음과 같다.

- 공공데이터는 더 이상 주력 해결책이 아니다.
- `KA-*` 운영시간 복구는 카카오 상세가 가장 효율적이다.
- 요금과 전화도 먼저 카카오 상세를 한 바퀴 돌린 뒤, 잔여분만 네이버 플레이스로 넘긴다.
- 면수는 별도 트랙으로 분리해서 다뤄야 한다.
- DDG는 카카오/네이버 구조화 소스를 모두 돈 뒤 마지막 잔여 처리에서만 쓴다.
- 우선순위는 당장은 결손량 기준, 이후 GA 결합 방식으로 고도화한다.

즉, 단기 실행 전략은 다음 한 줄로 요약된다.

`KA 노외 기본정보 결손 → 카카오 상세 전체 배치 → 잔여분만 네이버 플레이스 2차 보강 → 최종 잔여만 DDG fallback → 면수는 별도 프로젝트로 처리`
