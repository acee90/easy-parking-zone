# 주차장 통합 난이도 산출 알고리즘

본 문서는 시스템에 축적된 데이터(사용자 리뷰, 크롤링 블로그/카페, YouTube 댓글, 커뮤니티 댓글)를 종합하여 주차장별 난이도 점수를 산출하는 알고리즘을 설계합니다.

> **참고 사례**: 다이닝코드(빅데이터 맛집 점수), 망고플레이트(신뢰 리뷰어 가중치), Google(주차 난이도 ML)

---

## 1. 현황 및 문제

### 현재 시스템
- **난이도 점수**: `AVG(user_reviews.overall_score)` — 사용자 리뷰의 종합 평점 단순 평균 (1.0-5.0)
- **web_sources**: 56,000+ 크롤링 텍스트 데이터가 있으나 **난이도 산출에 미반영**
- **큐레이션**: hell/easy 태그 99개가 있으나 점수에 미반영
- **구조 속성**: 기계식/자주식, 총 면수 등 물리적 정보가 있으나 점수에 미반영

### 문제점
1. 사용자 리뷰가 없는 주차장은 난이도 정보 없음 (🅿️ 리뷰 없음)
2. 크롤링 데이터 56,000건이 난이도 산출에 활용되지 않음
3. "남산공원 소월로"처럼 블로그 정보는 많지만 직접 리뷰가 없는 주차장에 정보 공백
4. 큐레이션 된 "헬주차장"도 리뷰가 없으면 표시할 점수가 없음
5. 리뷰 1건 5점짜리가 리뷰 50건 평균 4.2보다 높게 표시됨 (표본 크기 무시)

---

## 2. 목표

- **출력**: 기존 난이도 체계 유지 (1.0-5.0 스케일)
- **입력 확대**: user_reviews + web_sources + curation + 구조 속성 모두 반영
- **호환성**: 기존 `getDifficultyIcon()`, `buildDifficultyCondition()` 변경 없이 동작

| 점수 | 아이콘 | 라벨 | 설명 |
|------|--------|------|------|
| 4.0-5.0 | 😊 | 초보 추천 | 넓고 여유로움 |
| 2.5-3.9 | 🙂 | 보통 | 일반적, 약간의 주의 |
| 1.5-2.4 | 💀 | 주의 | 좁거나 복잡, 경험 필요 |
| 1.0-1.4 | 💀💀 | 초보 비추 | 매우 좁거나 기계식 |

## 3. 데이터 소스 및 가중치

### 3.1. 소스 분류

| 소스 | 테이블 | 데이터 형태 | 기본 가중치 ($W$) |
|------|--------|-------------|-------------------|
| 사용자 리뷰 | `user_reviews` (source_type IS NULL) | 별점 1-5 (5개 항목) | **0.50** |
| 커뮤니티 리뷰 | `user_reviews` (source_type='clien' 등) | 별점 1-5 (AI 매칭) | **0.30** |
| 크롤링 텍스트 | `web_sources` (naver_blog, naver_cafe) | 텍스트 | **0.15** |
| 영상 댓글 | `web_sources` (youtube_comment) | 텍스트 | **0.15** |

> 블로그와 영상 댓글의 가중치를 동일하게 설정.
> 데이터가 없는 소스는 제외하고 **가중치를 비례 재분배**한다.

### 3.2. 가중치 재분배 예시

사용자 리뷰만 있는 경우:
```
W_user = 0.50 / 0.50 = 1.0  (사용자 리뷰가 100%)
```

크롤링 텍스트 + 영상 댓글만 있는 경우 (리뷰 없음):
```
W_blog = 0.15 / 0.30 = 0.50
W_youtube = 0.15 / 0.30 = 0.50
```

모든 소스가 있는 경우:
```
W_user=0.50, W_community=0.30, W_blog=0.15, W_youtube=0.15  (그대로)
```

## 4. 알고리즘 설계

### 4.1. 구조적 속성 사전 점수 (Structural Prior)

> Google 주차 난이도 ML 연구의 핵심 발견: 물리적 속성이 텍스트보다 강한 예측 변수

리뷰/텍스트 데이터가 없어도 DB에 이미 있는 물리적 속성으로 기본 난이도를 추정한다.
이 값은 베이지안 평균의 사전 확률(prior) $m$으로 사용된다.

```
기본값: 3.0 (중립)

감점 요인 (어려움):
  기계식 주차장         → -1.0
  총 면수 < 30         → -0.3
  지하 주차장           → -0.2
  curation_tag='hell'  → 직접 1.5 고정

가점 요인 (쉬움):
  노외 주차장           → +0.3
  총 면수 > 200        → +0.3
  무료 주차장           → +0.1 (접근성)
  curation_tag='easy'  → 직접 4.0 고정
```

산출: $m_{lot} = \text{clamp}(3.0 + \sum \text{보정값}, 1.0, 5.0)$

큐레이션 태그가 있으면 구조 속성 계산을 건너뛰고 앵커 값을 직접 사용한다.

### 4.2. 사용자/커뮤니티 리뷰 점수

`overall_score`의 가중 평균. 시간 감쇠를 적용한다.

$$S_{review} = \frac{\sum_{i} w_i \cdot d_i \cdot score_i}{\sum_{i} w_i \cdot d_i}$$

- 사용자 리뷰($w=1.0$), 커뮤니티 리뷰($w=0.6$), seed 리뷰($w=0.3$)
- $d_i$: 시간 감쇠 가중치 (4.6절 참조)
- 출력: 1.0 ~ 5.0

### 4.3. 텍스트 감성 분석 (크롤링/영상)

#### Step 1: 관련도 필터링 (Relevance Gate)

모든 텍스트가 주차 난이도 평가에 유용하지는 않다. 맛집/관광 후기에서 "주차 가능"만 언급한 글은 난이도 판단에 쓸 수 없다.

```
관련도 판별 기준:
  [높음] 주차 경험 직접 서술: "주차하기 좁다", "주차면이 넓어서", "나선형 진입"
  [중간] 주차 관련 언급: "주차장 있음", "주차 가능", "발렛 가능"
  [낮음] 주차 무관: 맛집 후기, 관광 정보 (주차 키워드 없음)
```

**관련도 점수** ($R$, 0.0-1.0):

| 조건 | 점수 |
|------|------|
| 주차 경험 키워드 2개 이상 포함 | 1.0 |
| 주차 경험 키워드 1개 포함 | 0.7 |
| "주차" 단어만 포함 (경험 서술 없음) | 0.3 |
| 주차 관련 키워드 없음 | 0.0 (제외) |

**주차 경험 키워드** (난이도 판단에 직접 관련):
```
긍정: 넓다/넓은/널널, 편하다/편한, 여유, 쉽다/쉬운, 초보, 추천, 평면, 자주리
부정: 좁다/좁은, 힘들다, 무섭다, 긁다/긁힐, 헬/hell, 골뱅이, 나선, 기계식,
     기둥, 사이드미러, 복잡, 급경사, 만차, 대기
```

#### Step 2: 부정어 처리

단순 키워드 카운팅의 한계를 보완한다.

```
부정어 패턴: 안/않/못/없/아닌/별로
적용 규칙: 부정어 + 2어절 이내 키워드 → 극성 반전

예시:
  "안 좁다" → 좁다(부정) 반전 → 긍정으로 처리
  "넓지 않다" → 넓다(긍정) 반전 → 부정으로 처리
  "주차하기 편하지 않은" → 편하다(긍정) 반전 → 부정
  "좁은 건 아닌데" → 좁다(부정) 반전 → 긍정
```

#### Step 3: IDF 가중 키워드 분석

모든 키워드를 동일 가중치로 카운팅하면 "주차"(어디든 등장) ≈ "골뱅이"(헬주차장에만 등장)가 되어 변별력이 떨어진다. 56K 코퍼스 기반 IDF를 사전 계산하여 진단력 높은 키워드에 자동으로 높은 가중치를 부여한다.

```
IDF(keyword) = log(전체 문서 수 / keyword 포함 문서 수)

예상 IDF 값:
  "주차"        → 0.1  (거의 모든 글에 등장, 저신호)
  "좁다"        → 0.6  (중간 빈도, 중간 신호)
  "골뱅이"      → 0.95 (극소수 글, 고신호 → 높은 가중치)
  "나선형"      → 0.92
  "사이드미러"   → 0.88
```

감성 키워드의 기여도 = 극성(+1/-1) × IDF 가중치

#### Step 4: 이모티콘/신조어 처리

한국어 텍스트에서 감성 정보를 담는 비표준 표현을 처리한다.

```
부정 신호: ㅠㅠ, ㅜㅜ, ㅎㄷㄷ(후덜덜), 지옥, 헬(hell)
긍정 신호: 👍, ㅎㅎ(맥락 의존), 꿀팁, 강추
강조 신호: ㄹㅇ(진짜), ㅈㄴ(매우), 진심 → 인접 키워드 가중치 1.5배
```

#### Step 5: 감성 점수 산출

텍스트별 감성 점수를 관련도 + IDF 가중하여 집계한다.

개별 텍스트의 감성 점수:
$$s_j = \frac{\sum (polarity_k \times IDF_k)}{\sum IDF_k}$$

- $polarity_k$: 키워드 $k$의 극성 (+1 또는 -1, 부정어 반전 적용 후)
- $IDF_k$: 키워드 $k$의 역문서빈도 가중치

소스별 통합 감성 점수 (관련도 + 시간 감쇠 가중):
$$S_{text} = \frac{\sum_{j} R_j \cdot d_j \cdot s_j}{\sum_{j} R_j \cdot d_j}$$

- $R_j$: 텍스트 $j$의 관련도 점수
- $d_j$: 시간 감쇠 가중치 (4.6절 참조)

#### Step 6: 1-5 스케일 변환 (부정성 편향 보정)

사람들은 어려운 주차 경험을 더 많이 글로 남기는 경향이 있다(부정성 편향).
중립점을 감성 0.0이 아닌 **+0.1에 매핑**하여 보정한다.

$$S_{text \rightarrow 5} = (S_{text} - 0.1) \times 2.0 + 3.0$$

| 감성 점수 | → 5점 스케일 | 의미 |
|-----------|-------------|------|
| +1.0 | 4.8 | 매우 긍정 |
| +0.5 | 3.8 | 긍정 |
| +0.1 | 3.0 | 중립 (보정 기준점) |
| -0.5 | 1.8 | 부정 |
| -1.0 | 0.8→1.0 | 매우 부정 (하한 클램프) |

### 4.4. 베이지안 통합 점수 산출

> 다이닝코드, 망고플레이트 등이 사용하는 핵심 패턴.
> 리뷰 1건의 5점이 리뷰 50건의 4.2점보다 높게 나오는 문제를 해결한다.

#### 1단계: 소스별 가중 평균 (Raw Score)

$$Score_{raw} = \sum_{k} W_k' \cdot S_k$$

- $k$: 활성 소스 (user, community, blog, youtube)
- $W_k'$: 재분배된 가중치 (합 = 1.0)
- $S_k$: 각 소스의 1-5 스케일 점수

#### 2단계: 유효 데이터량 계산

소스별 데이터 기여도를 정규화한다.

$$n_{eff} = n_{user} \times 1.0 + n_{community} \times 0.6 + n_{text(R \geq 0.7)} \times 0.2$$

- 사용자 리뷰 1건 = 유효 1.0건
- 커뮤니티 리뷰 1건 = 유효 0.6건
- 관련도 높은 텍스트 1건 = 유효 0.2건 (관련도 0.7 미만은 제외)

#### 3단계: 베이지안 평균

$$Score_{final} = \frac{C \cdot m_{lot} + n_{eff} \cdot Score_{raw}}{C + n_{eff}}$$

- $C = 5$: 신뢰 임계치 (유효 5건 이상이면 실제 데이터가 지배)
- $m_{lot}$: 구조적 사전 점수 (4.1절에서 산출)

**동작 원리**:
| 상황 | $n_{eff}$ | 결과 |
|------|-----------|------|
| 데이터 없음 | 0 | $m_{lot}$ (구조 속성 기반 추정) |
| 리뷰 1건 (5점) | 1.0 | $\frac{5 \times 3.0 + 1 \times 5.0}{6} = 3.33$ (과대평가 억제) |
| 리뷰 10건 (평균 4.2) | 10.0 | $\frac{5 \times 3.0 + 10 \times 4.2}{15} = 3.80$ (실제 평균 수렴) |
| 리뷰 50건 (평균 4.2) | 50.0 | $\frac{5 \times 3.0 + 50 \times 4.2}{55} = 4.09$ (실제 평균 지배) |

### 4.5. 신뢰도 등급

베이지안 평균이 신뢰도를 수학적으로 내재하지만, UI에서 사용자에게 점수 근거를 투명하게 보여주기 위해 등급을 표시한다.

| 등급 | 조건 | UI 표시 |
|------|------|---------|
| 확정 | $n_{eff} \geq 5$ (사용자 리뷰 3건 이상 등) | 점수만 표시 |
| 추정 | $1 \leq n_{eff} < 5$ | 점수 + "추정" 뱃지 |
| 참고 | $0 < n_{eff} < 1$ (텍스트만 소량) | 점수 + "참고" 뱃지 |
| 속성 기반 | 데이터 없음, 구조 속성으로 추정 | 점수 + "AI 추정" 뱃지 |
| 없음 | 구조 속성도 판별 불가 | 🅿️ 리뷰 없음 |

### 4.6. 시간 감쇠 (Time Decay)

주차장은 물리적 변화가 느리므로 반감기 365일 적용:

$$d(t) = 0.5^{경과일수 / 365}$$

| 경과 | 가중치 | 의미 |
|------|--------|------|
| 오늘 | 1.00 | 최신 |
| 6개월 | 0.71 | 약간 감소 |
| 1년 | 0.50 | 절반 |
| 2년 | 0.25 | 낮음 |
| 3년 | 0.13 | 거의 무시 |

리뷰 점수(4.2절)와 텍스트 감성 집계(4.3절) 모두에 적용한다.

## 5. 주의사항 및 엣지케이스

### 5.1. 블로그 SEO 오염

주차장 근처 맛집/카페 블로그 글에 "주차 가능", "주차 편리"가 보일러플레이트로 들어가는 경우가 많다. 실제 주차 경험 서술이 아니므로 관련도 게이트에서 차단해야 한다.

→ 관련도 0.3 이하("주차" 단어만 포함)는 감성 분석에서 **완전 제외**하는 것이 안전

### 5.2. 볼륨 편향

관광지/번화가 주차장은 블로그 언급이 많아 텍스트 소스에서 과대 대표될 수 있다.
→ 베이지안 평균이 자연스럽게 처리 (볼륨이 많으면 $n_{eff}$ 증가 → prior 영향 감소)
→ 추가로 텍스트 소스의 $n_{eff}$ 기여를 ×0.2로 제한하여 블로그 1건 = 리뷰 0.2건으로 환산

### 5.3. 감성 분석 실패 패턴

| 패턴 | 예시 | 대응 |
|------|------|------|
| 반어/풍자 | "주차 정말 편하죠 ㅎㅎ" (실제로는 어려움) | Phase 2에서 문맥 윈도우 확장 |
| 사투리 | "주차하기 빡시다" (= 힘들다) | 키워드 사전에 방언 추가 |
| 이중 부정 | "안 좁지 않다" (= 좁다) | 드문 패턴, Phase 1에서는 무시 |
| 조건부 평가 | "평일엔 괜찮은데 주말엔 좁음" | 부정 쪽으로 카운트 (보수적) |

### 5.4. 스팸/광고 리스크

주차장은 맛집과 달리 블로거 체험단이나 광고성 리뷰 인센티브가 거의 없다.
→ 스팸 탐지보다 **관련도 필터링이 더 중요** (Phase 1 집중)
→ 사용자 리뷰 어뷰징은 기존 rate-limit(ip_hash)으로 충분

## 6. 구현 계획

### Phase 1: 텍스트 감성 분석 파이프라인 + IDF 계산

**목표**: web_sources 텍스트를 분석하여 주차장별 감성 점수 산출

**작업**:
1. `scripts/compute-keyword-idf.ts` — 56K 코퍼스 기반 키워드 IDF 사전 계산
   - 결과: `data/keyword-idf.json` (키워드별 IDF 값)

2. `src/server/crawlers/lib/sentiment.ts` — 감성 분석 모듈 신설
   - 키워드 사전 (긍정/부정/부정어/이모티콘/신조어)
   - 관련도 판별 함수 (Relevance Gate)
   - 부정어 반전 로직
   - IDF 가중 감성 점수 변환
   - 부정성 편향 보정 (+0.1 중립점)

3. `scripts/compute-text-scores.ts` — 배치 스크립트
   - web_sources 텍스트에서 관련도($R$) + 감성($s$) 계산
   - 결과를 `web_sources.relevance_score`(관련도)와 `is_positive`(감성) 업데이트

4. 파일럿 검증 (**Phase 2 진행 전 필수**)
   - 이미 리뷰가 있는 주차장 50곳에서 텍스트 감성 점수 산출
   - 리뷰 평균과 텍스트 감성 점수의 피어슨 상관계수 → **목표 r ≥ 0.6**
   - 괴리가 큰 케이스 수동 확인 → 키워드/관련도 임계값 튜닝
   - r < 0.4이면 키워드 사전 확장 또는 관련도 게이트 강화 후 재검증

### Phase 2: 통합 점수 테이블 및 배치

**목표**: 베이지안 통합 점수를 사전 계산하여 저장

**작업**:
1. 마이그레이션: `parking_lot_stats` 테이블 신설
   ```sql
   CREATE TABLE parking_lot_stats (
     parking_lot_id TEXT PRIMARY KEY REFERENCES parking_lots(id),
     -- 구조 사전 점수
     structural_prior REAL,         -- 물리 속성 기반 사전 점수 (1-5)
     -- 소스별 점수
     user_review_score REAL,        -- 사용자 리뷰 가중 평균 (1-5)
     user_review_count INTEGER DEFAULT 0,
     community_score REAL,          -- 커뮤니티 리뷰 가중 평균 (1-5)
     community_count INTEGER DEFAULT 0,
     text_sentiment_score REAL,     -- 텍스트 감성 (1-5 변환)
     text_source_count INTEGER DEFAULT 0,
     -- 통합 결과
     n_effective REAL DEFAULT 0,    -- 유효 데이터량
     final_score REAL,              -- 베이지안 최종 난이도 (1-5)
     reliability TEXT,              -- 'confirmed'|'estimated'|'reference'|'structural'|'none'
     computed_at TEXT DEFAULT (datetime('now'))
   );
   ```

2. `scripts/compute-parking-stats.ts` — 통합 점수 배치 스크립트
   - 전체 주차장에 대해 4.1~4.5 알고리즘 적용
   - `parking_lot_stats` 테이블 갱신
   - 크론 또는 데이터 변경 시 실행

3. 쿼리 변경: `src/server/parking.ts`
   - 기존: `AVG(user_reviews.overall_score)` 동적 계산
   - 변경: `parking_lot_stats.final_score` 조인으로 교체
   - `reliability` 등급도 함께 반환

### Phase 3: UI 반영

**목표**: 신뢰도 뱃지 및 점수 출처 표시

**작업**:
1. `ParkingCard` / `ParkingDetailPanel`에 신뢰도 뱃지 표시
   - "추정", "참고", "AI 추정" 뱃지 — 툴팁으로 근거 설명
2. 상세 패널에 점수 구성 요소 시각화 (어떤 소스에서 어떤 점수가 나왔는지)
3. 기존 필터/마커 시스템은 변경 없음 (`final_score`가 1-5 스케일이므로)

### Phase 2+ (향후): 감성 분석 고도화

Phase 1의 키워드 기반 감성 분석이 r < 0.6이거나 개선 여지가 크면:
- 3문장 컨텍스트 윈도우로 확장 (주차 키워드 주변 ±1문장)
- KoBERT 파인튜닝 (주차 도메인 감성 분류, ~500 라벨링 샘플 필요)
- 반어/풍자 패턴 탐지 (KoCoSa 접근법 참고)

## 7. 검증 전략

### 상관관계 검증 (Phase 1 완료 시)
- 리뷰 3건 이상인 주차장에서 텍스트 감성 점수와 리뷰 평균의 피어슨 상관계수
- **목표**: r ≥ 0.6 (중간 이상 상관)
- r < 0.4이면 키워드 사전 또는 관련도 필터 재조정

### 큐레이션 일관성 (Phase 2 완료 시)
- hell 태그 주차장 83개의 `final_score` 분포 → 대부분 2.5 미만이어야 함
- easy 태그 주차장 16개의 `final_score` 분포 → 대부분 3.5 이상이어야 함
- 위반 케이스는 수동 검토 → 구조 사전 점수 또는 감성 분석 조정

### 커버리지 비교 (Phase 2 완료 시)
- 기존 방식(리뷰 평균만) vs 새 방식(베이지안 통합)의 커버리지 비교
- 점수가 새로 부여되는 주차장 수 확인 (현재 "리뷰 없음" → 점수 있음)
- 등급 변동이 생기는 주차장 목록 → 수동 검토로 합리성 확인

### 베이지안 파라미터 튜닝 (Phase 2 완료 시)
- $C$ (신뢰 임계치) 값을 3, 5, 7, 10으로 변경하며 결과 비교
- 큐레이션 일관성 + 사용자 리뷰 수렴 속도 기준으로 최적 $C$ 선정

## 8. 기대 효과

- **커버리지 확대**: 리뷰 없는 주차장도 크롤링/구조 데이터로 난이도 표시 가능
- **정확도 향상**: 다중 소스 교차 검증 + 베이지안 보정으로 소표본 편향 제거
- **cold-start 해결**: 구조 속성 + 큐레이션 앵커로 신규 주차장도 즉시 점수 부여
- **호환성**: 기존 1-5 스케일 유지로 UI/필터 변경 최소화
- **투명성**: 신뢰도 등급으로 점수 근거를 사용자에게 공개
