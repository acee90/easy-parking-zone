# 주차장 AI 요약 배치 생성

## 개요

서브에이전트가 한 번 호출될 때마다 **N건의 주차장 AI 요약을 생성**하고 D1에 저장합니다.
`ANTHROPIC_API_KEY` 불필요 — Claude 자신이 직접 요약을 생성합니다.

## 워크플로우

### Step 1: 데이터 추출

```bash
cd /Users/junhee/Documents/projects/parking-map/main
bun run scripts/extract-lots-for-summary.ts --limit=20
```

`summary_batch.json` 생성 확인. 0건이면 "모든 대상 처리 완료" 보고 후 종료.

---

### Step 2: AI 요약 생성 (직접)

`summary_batch.json`을 읽고, 각 주차장에 대해 아래 JSON을 직접 생성합니다.

**출력 형식 (주차장당):**
```json
{
  "id": "주차장 ID",
  "summary": "전체 특징 2~3문장 (120~180자). 진입 난이도·주차면 넓이·요금·혼잡 시간대 위주.",
  "tip_pricing": "요금 구조·할인·무료 조건 1~2문장. 근거 없으면 null.",
  "tip_visit": "진입 경로·혼잡 시간대·주의사항 1~2문장. 근거 없으면 null.",
  "tip_alternative": "근처 대안 주차장·대중교통 연계 1~2문장. 근거 없으면 null."
}
```

**생성 규칙:**
- 반드시 경어체(~습니다, ~합니다)만 사용, 평서체(~다, ~이다) 금지
- "AI가 분석했다" 같은 메타 표현 금지
- 과장, 이모지, 마크다운 금지
- 모순 의견 → "대체로 ~하지만 ~라는 의견도 있습니다" 형식
- 근거가 빈약한 필드 → null

전체 결과를 `summary_results.json`으로 저장합니다.

---

## 품질 평가 (Eval)

생성 후 반드시 eval 스크립트를 실행해 품질을 확인합니다.

```bash
cd /Users/junhee/Documents/projects/parking-map/main
bun run scripts/eval-summary-quality.ts
bun run scripts/eval-summary-quality.ts --verbose  # 전문 출력
```

### 평가 항목 (E1~E6)

| 항목 | 기준 | 통과 조건 |
|------|------|-----------|
| E1 경어체 | 모든 필드가 ~습니다/합니다로 끝나는지 | 평서체 없음 |
| E2 길이 | summary 길이 | 100~220자 (권장 120~180자) |
| E3 금지표현 | "AI가", "데이터에 따르면", 이모지 | 없음 |
| E4 소스관련성 | 주차장 이름 직접 언급 비율 | 30% 이상 권장 |
| E5 null 규율 | null 필드인데 소스에 관련 키워드 있는지 | 불일치 없음 |
| E6 중복 | summary와 tip 간 동일 어절 반복 | 5개 미만 |

### 등급 기준

- **A**: E1~E3, E5, E6 모두 통과 + 소스관련성 50% 이상
- **B**: 5개 통과 + 소스관련성 30% 이상
- **C**: 3~4개 통과
- **F**: 2개 이하 통과

### 개선 기준

- **소스관련성 30% 미만**: 해당 주차장은 생성 가치 낮음 → 배치에서 제외 검토
- **E1 실패**: 프롬프트 규칙 위반 → 재생성
- **E3 실패**: 즉시 재생성 (SEO 품질 직결)

## 생성 방식 비교 (실험 완료)

### 실험 결과 요약 (5개 lot 대상)

| lot | A 등급 | B 등급 | 실질 차이 |
|-----|--------|--------|-----------|
| 강경 대흥리 | C | C | 없음 |
| 성류굴 | C | C | **있음** — B가 유/무료 정보를 더 명확히 정리 |
| 강경 산양1리 | C | C | 없음 |
| 풍기읍 | F | F | 없음 |
| 추풍령 | C | C | 없음 |

### 핵심 발견

**Eval 점수로는 A/B 차이를 검출할 수 없다.** E1~E6 모두 형식·규칙 준수를 측정할 뿐, 내용 정확도나 판단 품질을 측정하지 않기 때문이다.

**성류굴 사례 (유일한 실질 차이):**
- A: 13개 소스에서 "무료"(11건) vs "정문 유료"(1건) 언급이 뒤섞여 → "모순 있어 확인 필요"로 소극적 서술
- B: Map step에서 비율 파악(11:1) → "공영주차장은 무료, 정문 별도 유료 가능성" 식으로 더 명확한 판단

**소스가 thin할 때는 방법 차이가 없다.** 4개 lot은 이 주차장을 직접 다룬 소스가 없어 map/reduce를 해도 추출할 정보가 없다.

### 결론

- **Method A** 유지 (소스 10건 이하, 또는 직접 언급 소스 30% 미만)
- **Method B** 적용 (소스 10건 이상이고 직접 언급 소스가 많은 경우 — 대형 관광지 주차장 등)
- **근본 문제**: 한국 블로그 web_sources 대부분은 식당/카페 글에서 "근처 공영주차장 이용 가능" 식으로 주차장을 언급하므로 소스 자체의 parking-specificity가 낮음. 방법론보다 소스 품질이 더 중요한 변수.

## Eval 항목 비판 및 개선안

### 현행 E1~E6 신호 품질 평가

| 항목 | 신호 품질 | 문제 |
|------|-----------|------|
| E1 경어체 | 중 | **구현 버그**: `[습합입]니다` 정규식이 "됩니다", "드립니다" 미포함 → 거짓 양성 다수 |
| E2 길이 | 저 | 소스가 얇으면 내용도 자연히 짧아짐. 길이로 품질 측정 불가 |
| E3 금지표현 | 고 | 실제로 유효. SEO 품질 직결 |
| E4 소스관련성 | 고 | 핵심 메트릭이지만 **4글자 prefix 매칭이 너무 엄격** → 성류굴처럼 정식 명칭이 소스에 있어도 3/13만 인식 |
| E5 null 규율 | 저 | 거짓 양성 많음: 소스가 다른 주차장 요금 언급해도 keywords 감지 → null이 맞는데 실패 판정 |
| E6 중복 | 저 | 5건 모두 통과. 차별화 능력 없음 |

### 추가 제안 항목 (E7, E8)

**E7 정보밀도** (중~고 신호)
- summary 내 주차 관련 키워드(무료/유료/요금/면수/진입/혼잡/층/분) 포함 비율
- 기준: 2개 이상이면 통과 (식당 후기 복붙을 탐지)

**E8 할루시네이션 리스크** (고 신호, 수동 체크)
- summary/tip에 포함된 숫자·고유명사가 소스에 실제로 있는지 역추적
- 자동화 어려움 → 표본 수동 검토 체크리스트로 운영

### E1 버그 수정 방향
```ts
// 현재 (버그): 됩니다, 드립니다 등 누락
const POLITE = /[습합입]니다[.]?$/;

// 수정안: ㅂ니다 패턴 전체 허용
const POLITE = /[가-힣]니다[.]?$/;
```
